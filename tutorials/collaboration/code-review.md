# 代码审查指南

参考来源：[GitHub Docs 入门](https://docs.github.com/zh/get-started)。

## 什么时候需要 Review？

代码审查用于在合并前发现逻辑错误、风格不一致、缺少测试、权限风险和文档缺口。它也是团队共享上下文的好机会。

## 提交者清单

1. PR 只解决一个主题。
2. 标题说明意图，描述写清楚背景、变更和测试。
3. 把大改动拆成多个小提交或多个 PR。
4. 本地运行测试和构建。
5. 对审查意见逐条回复，修改后说明处理方式。

## 审查者清单

| 检查点 | 说明 |
| --- | --- |
| 行为正确性 | 是否真的解决问题，是否引入边界条件错误 |
| 可维护性 | 命名、结构、重复代码、复杂度是否合理 |
| 测试 | 是否覆盖核心路径和失败路径 |
| 安全 | 是否泄露 secret、扩大权限、绕过校验 |
| 文档 | 用户可见行为是否需要更新 README 或教程 |

## Review 结论

- **Comment**：只留言，不阻塞合并。
- **Approve**：同意合并。
- **Request changes**：必须修改后再合并。
